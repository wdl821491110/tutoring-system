README 中已提到审核要点，但代码中**未落地**：

| 缺失项 | 严重程度 | 说明 |

```javascript
        // 引导用户同意隐私协议
        wx.requirePrivacyAuthorize({ success: () => {}, fail: () => {} });
      }
    }
  });
}
```

### 3. 数据直接变异 Bug — setData 前修改了 data 对象

在 `schedules.js` 和 `records.js` 中有多处直接修改 `this.data.form` 而不先拷贝：

```javascript
// ❌ 危险：直接修改了 this.data.form
onFormChange: function(e) {
  var f = this.data.form;
  f[e.currentTarget.dataset.field] = e.detail.value; // 直接修改原对象！
  this.setData({ form: f });
}
```

虽然 `setData({ form: f })` 仍能触发渲染，但 `f` 与 `this.data.form` 是**同一个引用**——后续 `this.data.form` 比较会出问题，且不符合单向数据流原则。

**影响范围**: `schedules.js` L61, L63-69, L71 / `records.js` L42, L44, L52

**修复**: 始终创建新对象
```javascript
onFormChange: function(e) {
  var f = Object.assign({}, this.data.form);
  f[e.currentTarget.dataset.field] = e.detail.value;
  this.setData({ form: f });
}
```

### 5. 三个并发的 API 调用无错误处理

`schedules.js` L46-59 和 `records.js` L29-40 中，三个 API 是嵌套调用的（非并行），且任一失败都会导致 `wx.hideLoading()` 不执行，用户看到永久 loading：

```javascript
// ❌ 嵌套调用，任一失败则 loading 不消失
app.api('/api/students?status=active').then(function(s) {
app.api('/api/courses').then(function(c) {
app.api('/api/enrollments?status=active').then(function(e) {
  wx.hideLoading();
  // ...
});});});
```

**修复**: 使用 `Promise.all` 并行调用 + 错误处理
```javascript
Promise.all([
  app.api('/api/students?status=active'),
  app.api('/api/courses'),
  app.api('/api/enrollments?status=active')
]).then(function([s, c, e]) {
  wx.hideLoading();
  // ...
}).catch(function() {
  wx.hideLoading();
  wx.showToast({ title: '加载失败', icon: 'none' });
});
```

## 🟠 重要问题（影响体验或长期维护）

### 6. 项目无模块化结构

所有工具函数（API 请求、Token 管理、角色判断）全部堆在 `app.js` 中。

```
❌ 现状:                  ✅ 建议:
app.js (127行)            app.js (~30行)
├── api()                 utils/
├── isAdmin()             ├── request.js     # 统一请求 + 拦截器
├── isTeacher()           ├── auth.js        # Token/Login 管理
└── onLaunch              └── storage.js     # 缓存工具
```

**建议**: 创建 `utils/` 目录，将请求层和认证层独立出来。当前 `app.js` 的 `api()` 方法已有 27 行，再增加重试/缓存逻辑会更难维护。

### 7. 使用 `var` 而非 `let/const`

项目配置 `es6: true`，所有 `.js` 仍使用 `var`。这会导致变量提升、作用域问题：

```javascript
// schedules.js L15 — statuses 数组在每次调用 load() 时重复创建
var m = ['','scheduled','completed','cancelled'];
```

**建议**: 全局替换为 `let`/`const`，将常量提取到页面顶部或 data 外。

`pages/tabbar/` 下完整实现了一套自定义 TabBar 组件（使用 emoji 图标），但 `app.json` 中未配置 `"tabBar": { "custom": true }`。目前使用的是原生 TabBar。

### 9. 登录页作为首页 — 可能的闪现问题

`app.json` 中 `pages/login/login` 排在第一位，意味着用户每次打开小程序都会先进入登录页。虽然各 TabBar 页面的 `onShow` 有 token 检查，但这会导致：
- 已登录用户看到登录页的短暂闪现 → 然后跳转到首页
- 冷启动时登录页渲染 → token 检查 → reLaunch 到首页，造成不必要的页面开销

**建议**: 将 `pages/index/index` 放在第一，在 `app.js` 的 `onLaunch` 中做统一鉴权：

```javascript
// app.js
onLaunch: function() {
  var token = wx.getStorageSync('token');
  if (!token) {
    // 重定向到登录页
    wx.reLaunch({ url: '/pages/login/login' });
  } else {
    this.globalData.token = token;
    this.globalData.user = wx.getStorageSync('user');
  }
}
```

> **注意**: TabBar 页面不能用 `wx.redirectTo` 跳转到非 TabBar 页。已登录用户的冷启动不受影响——`index` 在 `pages` 第一位则默认打开首页。

## 🟡 一般问题（体验优化）

### 11. 无加载骨架屏/加载态

除登录页外，所有页面无 loading 状态。数据加载期间页面显示为空（或旧数据），用户体验差。

| 页面 | 加载态 | 问题 |
|------|-------|------|
| 首页 | ❌ | stats 为空对象，统计卡片显示 0 |
| 学生 | ❌ | 列表短暂为空 + 空状态提示 |
| 排课 | ❌ | 同上 |
| 课时 | ❌ | 同上 |
| 个人 | ❌ | 用户信息短暂为空 |

**建议**: 至少增加 `loading: true` 状态 + WeUI 的 `mp-loading` 组件，或在统计卡片处放置骨架屏。

---

### 12. 每次 onShow 强制全量刷新

所有 TabBar 页面的 `onShow` 都直接调用 `load()`，无任何缓存策略。用户切换 tab 就会有网络请求。

**建议**: 实现简单的过期缓存：
```javascript
load: function(force) {
  var cache = this._cache;
  if (!force && cache && Date.now() - cache.time < 30000) {
    this.setData(cache.data);
    return Promise.resolve();
  }
  return app.api('/api/dashboard').then(function(res) {
    var data = { stats: res.data || {} };
    this._cache = { time: Date.now(), data: data };
    this.setData(data);
  });
}
```

3. **工程化**（P2-P3）：模块化、代码风格、缓存策略

建议**优先处理 P0 和 P1 项**（合计约 5 小时工作量），即可准备提审。P2/P3 项可在后续迭代中逐步优化。
