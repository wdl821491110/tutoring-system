## 四、架构/设计层面优化建议

### 4.1 路由与 TabBar

| 当前 | 建议 | 理由 |
|------|------|------|
| TabBar 只有 5 个入口 | 增加"课程"Tab 或使用二级页面 | 课程管理是高频操作，目前完全无法在手机上操作 |
| 无权限控制 Tab 显隐 | 按角色动态控制 Tab | admin/teacher 看到的 Tab 应不同 |

### 4.2 数据层

| 问题 | 建议 |
|------|------|
| 无全局数据缓存/状态管理 | 引入简单的全局 data store（或利用 `app.globalData` 扩展），减少重复 API 请求 |
| 每页独立 30s 缓存逻辑各自实现 | 抽象为 `utils/cache.js` 统一缓存策略 |
| 无请求重试 | request.js 增加网络错误自动重试（1-2次） |
| 无请求并发控制 | 批量操作需注意 `wx.request` 并发限制 |

### 4.3 安全

| 问题 | 建议 |
|------|------|
| token 明文存 Storage | 考虑加密存储或使用微信安全能力 |
| 无操作日志 | 关键操作（删除/消课/撤销）增加操作审计 |
| `parent` 角色无专属页面 | 如有家长端需求，需为 parent 角色设计只读/受限视图 |

### 4.4 代码质量

| 问题 | 位置 | 建议 |
|------|------|------|
| `records.js` 414 行，过长 | 拆分为按日期/按学生/备注三个模块 |
| `students.js` / `schedules.js` / `records.js` 重复的表单逻辑 | 抽象 `utils/form-helper.js` |
| 硬编码字符串多 | 抽取常量文件 `utils/constants.js` |
| validator 仅 4 种校验 | 扩展为通用 schema validator |

### 4.5 微信能力未利用

| 能力 | 现状 | 建议 |
|------|------|------|
| `wx.login` + openid | ❌ 未使用 | 实现微信静默登录，绑定系统用户 |
| 订阅消息 | ❌ | 上课提醒 / 课时不足通知 |
| 手机号授权 | ❌ | 快速验证家长手机号 |
| 云开发数据库 | ❌ 使用 CloudRun + SQLite | 若数据量增长可考虑迁移至 CloudBase 数据库以支持实时同步 |
